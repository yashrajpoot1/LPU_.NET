C# Coding Best Practices

A practical, real‑world guide for writing readable, maintainable, and professional C# code.

What you will learn: naming conventions, clean code habits, correct exception handling, async rules, LINQ best practices, performance basics, security hints, and a daily checklist.

Beginner-friendly Real-world examples Copy-paste ready code

1) What are “best practices”?

Best practices are commonly accepted patterns that make code easier to read, safer to change, and less likely to produce bugs in production.

Golden Rules: Prefer clarity over cleverness, keep responsibilities small, handle errors intentionally, and write code as if someone else will maintain it (because they will!).

Readable

Names explain intent, functions are small, and the code “tells the story”.

Maintainable

Easy to test, easy to change, and less risky for future updates.

2) Naming Conventions (C# style)

Consistent naming is the fastest way to make your code look professional. Below are widely used conventions in modern C# projects.

Item Convention Example Why it matters
Class / Struct / Record PascalCase OrderService, CustomerAddress Types stand out clearly.
Method PascalCase (verbs) CalculateTotal(), SaveAsync() Methods describe an action.
Property PascalCase (nouns) TotalAmount, CreatedAt Properties represent data.
Local variable / Parameter camelCase orderId, discountRate Easy to differentiate from types.
Private field _camelCase _logger, _dbContext Clearly indicates a field.
Interface Prefix with I IOrderRepository Immediately recognized as a contract.
Enum PascalCase (singular) OrderStatus Represents one value at a time.
Async methods Suffix with Async GetOrderAsync() Signals that it returns a Task.
Constant PascalCase MaxRetryCount Same style as other members in C#.
Good naming example (real-world: order processing) Try It Yourself (paste into a console app) 
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(Guid orderId, CancellationToken cancellationToken);
    Task SaveAsync(Order order, CancellationToken cancellationToken);
}

public sealed class OrderService
{
    private readonly IOrderRepository _orderRepository;

    public OrderService(IOrderRepository orderRepository)
    {
        _orderRepository = orderRepository;
    }

    public async Task<decimal> CalculateTotalAsync(Guid orderId, CancellationToken cancellationToken)
    {
        var order = await _orderRepository.GetByIdAsync(orderId, cancellationToken);

        if (order is null)
            throw new InvalidOperationException($"Order not found: {orderId}");

        var totalAmount = order.Items.Sum(i => i.UnitPrice * i.Quantity);
        return totalAmount;
    }
}
Avoid: meaningless names like temp, data, obj, or abbreviations like calcTot.

3) Project & Folder Structure

Good structure prevents “spaghetti projects”. A simple and common approach:

Suggested structure (for API / Service projects)
MyApp/
  MyApp.Domain/           // Entities, value objects, domain rules
  MyApp.Application/      // Use-cases, business services, DTOs
  MyApp.Infrastructure/   // EF Core, external APIs, repositories
  MyApp.WebApi/           // Controllers, middleware, DI, endpoints
  MyApp.Tests/            // Unit/integration tests
Tip: Keep “pure business rules” away from frameworks. Your business logic should not depend on ASP.NET Core, EF Core, etc.

4) Clean Code Rules

4.1 Keep methods small (single responsibility)

A method should do one thing well. If it’s doing 5 things, split it.

Bad vs Good: validation + business logic
// ❌ BAD: too many responsibilities
public async Task PlaceOrderAsync(OrderRequest request)
{
    if (request == null) throw new ArgumentNullException(nameof(request));
    if (request.Items == null || request.Items.Count == 0) throw new Exception("No items");

    // calculate total
    decimal total = 0;
    foreach (var item in request.Items)
        total += item.Price * item.Quantity;

    // save to database
    await _db.SaveAsync(new Order { Total = total });
}

// ✅ GOOD: clear, testable, reusable
public async Task PlaceOrderAsync(OrderRequest request, CancellationToken ct)
{
    ValidateOrderRequest(request);

    var order = CreateOrderFrom(request);
    await _orderRepository.SaveAsync(order, ct);
}

private static void ValidateOrderRequest(OrderRequest request)
{
    if (request is null) throw new ArgumentNullException(nameof(request));
    if (request.Items is null || request.Items.Count == 0)
        throw new ArgumentException("Order must have at least one item.", nameof(request.Items));
}

4.2 Prefer guard clauses (reduce nesting)

Guard clause example
// ❌ BAD: deep nesting
if (user != null)
{
    if (user.IsActive)
    {
        if (user.EmailConfirmed)
        {
            // ...
        }
    }
}

// ✅ GOOD: early exits
if (user is null) return;
if (!user.IsActive) return;
if (!user.EmailConfirmed) return;

// ... main logic

4.3 Prefer nameof() in exceptions

nameof() keeps error messages safe during refactor
public void SetDiscount(decimal discountRate)
{
    if (discountRate < 0 || discountRate > 1)
        throw new ArgumentOutOfRangeException(nameof(discountRate), "Discount must be between 0 and 1.");
}

4.4 Use “var” only when it improves readability

Rule of thumb: Use var when the right side clearly shows the type (e.g., new List<int>()), otherwise write the explicit type.

5) Exceptions & Error Handling

5.1 Do not catch exceptions you can’t handle

Catching and ignoring errors makes bugs invisible. If you catch, do something meaningful: log, convert, retry, or return a safe result.

Bad vs Good: exception handling
// ❌ BAD: Swallows error, hides production bugs
try
{
    await _paymentGateway.ChargeAsync(amount);
}
catch
{
    // do nothing
}

// ✅ GOOD: log + throw a domain-specific exception
try
{
    await _paymentGateway.ChargeAsync(amount);
}
catch (TimeoutException ex)
{
    _logger.LogWarning(ex, "Payment timeout for OrderId={OrderId}", orderId);
    throw new PaymentFailedException("Payment timed out. Please try again.", ex);
}

5.2 Prefer specific exceptions

Use ArgumentException, ArgumentNullException, ArgumentOutOfRangeException, or custom exceptions for business rules.

Best practice: Keep exception messages user-friendly at the UI boundary, and detailed logs in the server logs.

6) Async/Await Best Practices

6.1 Never use async void (except event handlers)

Correct async signatures
// ❌ BAD
public async void Save() { ... }

// ✅ GOOD
public async Task SaveAsync(CancellationToken ct) { ... }

6.2 Support CancellationToken in library / service methods

CancellationToken example
public async Task<IReadOnlyList<Order>> SearchOrdersAsync(string query, CancellationToken ct)
{
    if (string.IsNullOrWhiteSpace(query))
        return Array.Empty<Order>();

    return await _orderRepository.SearchAsync(query, ct);
}

6.3 Avoid blocking calls (no .Result / .Wait())

Why: Blocking async calls can cause deadlocks and performance issues, especially in UI apps and ASP.NET environments.
Bad vs Good: async usage
// ❌ BAD
var order = _repo.GetByIdAsync(orderId).Result;

// ✅ GOOD
var order = await _repo.GetByIdAsync(orderId, ct);

6.4 Use HttpClient correctly

In real projects, prefer IHttpClientFactory (ASP.NET Core) or a shared, long-lived HttpClient.

7) LINQ & Collections

7.1 Prefer clear LINQ over complex queries

Readable LINQ (Orders)
var recentHighValueOrders = orders
    .Where(o => o.CreatedAt >= DateTime.UtcNow.AddDays(-7))
    .Where(o => o.TotalAmount >= 5000)
    .OrderByDescending(o => o.TotalAmount)
    .Take(10)
    .ToList();

7.2 Avoid multiple enumeration

Tip: If you use the same query many times, materialize it once using ToList() or ToArray().

8) Performance & Memory

8.1 Use using/await using for IDisposable

Dispose resources correctly
// ✅ GOOD
using var stream = File.OpenRead(path);
using var reader = new StreamReader(stream);
var content = await reader.ReadToEndAsync();

8.2 Prefer StringBuilder for many concatenations

StringBuilder example
var sb = new StringBuilder();
foreach (var item in items)
{
    sb.AppendLine(item);
}
var output = sb.ToString();

8.3 Avoid premature optimization

Rule: Optimize after measuring. Use benchmarks/profilers before rewriting code.

9) Security & Safety

9.1 Never build SQL with string concatenation

SQL injection safety (concept)
// ❌ BAD (dangerous)
var sql = "SELECT * FROM Users WHERE Email = '" + email + "'";

// ✅ GOOD: use parameters (ADO.NET / Dapper / EF Core)
command.Parameters.AddWithValue("@Email", email);

9.2 Validate inputs at boundaries

Validate user input at API/UI boundaries (controllers, endpoints) and validate business rules inside services.

9.3 Prefer immutable data when possible

Modern C#: consider record types for DTOs and message models to reduce accidental mutation.

10) Testing & Maintainability

10.1 Write code that is easy to test

Use dependency injection, small methods, and avoid static/global state.

Test-friendly design (service + interface)
public interface IClock
{
    DateTime UtcNow { get; }
}

public sealed class SystemClock : IClock
{
    public DateTime UtcNow => DateTime.UtcNow;
}

public sealed class SubscriptionService
{
    private readonly IClock _clock;

    public SubscriptionService(IClock clock)
    {
        _clock = clock;
    }

    public bool IsExpired(DateTime expiresAtUtc)
        => expiresAtUtc <= _clock.UtcNow;
}

10.2 Use Arrange-Act-Assert in unit tests

AAA test layout (illustration)
// Arrange
var clock = new FakeClock(DateTime.Parse("2025-12-19T00:00:00Z"));
var svc = new SubscriptionService(clock);

// Act
var expired = svc.IsExpired(DateTime.Parse("2025-12-18T00:00:00Z"));

// Assert
Assert.True(expired);

11) Quick Checklist (Daily Use)

Before you commit code

  • Names are meaningful (no data, temp, obj).
  • Methods are small and do one thing.
  • Validation uses guard clauses.
  • Exceptions are specific and not swallowed.
  • Async methods use Task, not async void.
  • CancellationToken is passed where appropriate.
  • LINQ is readable; avoid complex one-liners.
  • Disposables are disposed via using.
  • No secrets in code (keys, passwords, connection strings).
  • Run tests + format code before push.
