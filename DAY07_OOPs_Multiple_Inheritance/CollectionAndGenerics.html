Creating and Using Generic Classes in C#

In C#, type parameters help you build reusable, strongly-typed components. This page explains generic classes, constraints, and real-world patterns (like repositories).

Note: Generics are verified at compile-time (before running).

Generic Classes
Generic classes are “templates” for classes. You define them once, and use them with many types. They reduce repeated code, remove risky casting, and keep APIs consistent. Generics are supported by the CLR, so they are fast and safe.
Why Generics?

Without generics, you often store values as object and then cast them back—this is error-prone. With type safety, your code becomes cleaner and easier to maintain. Generics also improve reusability: one class works for many types.

ASCII Flow (Why Generics) 
Without Generics:         With Generics:
object box = 10;          Box<int> box = new Box<int>(10);
(int)box  ✅/❌            box.Value (int) ✅ always safe
Try It Yourself 
// Non-generic example (risk of casting bugs)
object value = "10";
// int n = (int)value; // ❌ InvalidCastException at runtime

// Generic idea: Value is always the right type
class Box<T> { public T Value; public Box(T v){ Value = v; } }

Real-world use case: A reusable “Result<T>” wrapper for API responses (Success + Data + ErrorMessage).

Beginner Tip: Start with one type parameter: T. Keep it simple.
Advanced Note: Generics reduce boxing/unboxing and avoid runtime casts in many scenarios.

Create a Generic Class (T)

A generic class declares a type parameter like T. When you create an instance, you “fill in” the real type: Box<int>, Box<string>, etc. The compiler enforces correct usage, so you don’t need manual casting.

ASCII Diagram (Generic Class Template) 
Box<T>  (template)
   |
   +-- Box<int>    Value: int
   +-- Box<string> Value: string
public class Box<T>
{
    public T Value { get; }

    public Box(T value)
    {
        Value = value;
    }
}

// Usage
var intBox = new Box<int>(99);
var strBox = new Box<string>("Hello");

Real-world use case: A Cache<T> that stores typed objects and returns them safely.

Beginner Tip: Name type parameters clearly: TKey, TValue, TItem.
Advanced Note: Prefer immutability (read-only properties) for safer generic containers.

Multiple Type Parameters (TKey, TValue)

Some designs need more than one type parameter. Example: a key-value pair uses TKey for the key and TValue for the value. This is common in dictionaries, caches, and repositories.

ASCII Diagram (Key → Value) 
TKey ----> TValue
"IN-1001" ----> Invoice
101        ----> Customer
public class KeyValueStore<TKey, TValue>
{
    private readonly Dictionary<TKey, TValue> _data = new();

    public void Add(TKey key, TValue value) => _data[key] = value;
    public TValue Get(TKey key) => _data[key];
}

// Usage
var store = new KeyValueStore<string, int>();
store.Add("Apples", 5);

Real-world use case: ProductCode (string) → StockCount (int) mapping in inventory.

Beginner Tip: Keep APIs minimal: Add(), Get(), TryGet().
Advanced Note: Consider TryGet to avoid exceptions for missing keys.

Generic Constraints (where T : ...)

Constraints restrict what types can replace T. This lets you safely use members like parameterless constructors or base-class methods. Constraints improve correctness and developer experience.

ASCII Cheat Sheet 
where T : class      // reference types only
where T : struct     // value types only
where T : new()      // must have parameterless ctor
where T : BaseType   // must inherit BaseType
public class Factory<T> where T : new()
{
    public T Create() => new T();
}

public class Customer
{
    public Customer() { }
}

var f = new Factory<Customer>();
Customer c = f.Create();

Real-world use case: A factory that creates typed DTO objects in a mapping
